De laboratorio a producción: arquitectura, instalación y requerimientos

Cómo pasar de las 3 sucursales simuladas en Docker a una red real de puntos de venta — qué instalar en cada sucursal, cómo queda la arquitectura completa, y qué se necesita (infraestructura, red, seguridad, equipo) para llevarlo a producción.

Basado en: debezium-lab (validado con 3 sucursales simuladas) Escenario objetivo: 50+ sucursales, conectividad mixta

1. Dos formas de instalar Debezium en las sucursales

Antes de la arquitectura general hay que decidir esto: ¿quién se conecta hacia quién? Es la decisión que más condiciona todo lo demás — seguridad de red, qué hay que instalar en cada sucursal, y qué tan bien tolera el sistema una sucursal con internet inestable.

  Opción A — Kafka Connect centralizado Opción B — Debezium Server por sucursal
Cómo funciona Un clúster de Kafka Connect en el centro corre un conector por sucursal (es el modelo exacto del laboratorio, con las 3 sucursales en la misma red Docker) Un proceso liviano (Debezium Server) instalado junto a cada MySQL de sucursal, que empuja los eventos directo al Kafka central
Dirección de la conexión El centro se conecta hacia cada sucursal — necesita una ruta de entrada a cada MySQL de POS Cada sucursal se conecta hacia afuera, al Kafka central — no requiere abrir ningún puerto de entrada
Firewall / NAT en la sucursal Requiere abrir un puerto entrante permanente por sucursal — fricción de seguridad real con 50+ ubicaciones Tráfico saliente por HTTPS/TLS estándar — compatible con el firewall típico de una sucursal de retail sin cambios especiales
Ante una sucursal caída o sin internet El conector de esa sucursal falla en el centro; requiere que el centro "alcance" la sucursal para reconectar El agente local sigue leyendo el binlog aunque no haya salida a internet, y reintenta el envío solo — tolera mejor una conexión intermitente
Recomendación Opción B para el escenario real (50+ sucursales, conectividad mixta) recomendada — Opción A solo tiene sentido si todas las sucursales ya están en una VPN privada confiable hacia el centro.
El resto de este documento asume Opción B. Si el negocio ya tiene una VPN corporativa madura conectando todas las sucursales, la Opción A es más simple de operar (un solo lugar para administrar todos los conectores) y el laboratorio ya la modela tal cual — avísame y ajusto esta propuesta.

2. Arquitectura objetivo

Cada sucursal captura y empuja sus propios cambios; el centro los recibe, los consolida, y expone señales de salud reales — nada de esto depende de que alguien esté mirando una terminal.

┌─ Sucursal 1 ─────────────────┐ │ MySQL POS │ │ └─ binlog ROW + GTID │ │ └─ usuario dedicado │ │ Debezium Server (agente) │──┐ └───────────────────────────────┘ │ TLS, saliente ┌─ Sucursal 2 ─────────────────┐ │ │ MySQL POS + Debezium Server │──┼──────────┐ └───────────────────────────────┘ │ │ ┌─ Sucursal N (50+) ───────────┐ │ ▼ │ MySQL POS + Debezium Server │──┘ ┌───────────────────────┐ └───────────────────────────────┘ │ Kafka (clúster, 3+ brokers│ │ factor de replicación ≥3)│ └───────────┬───────────┘ │ ┌─────────────────┼─────────────────┐ ▼ ▼ ┌─────────────────────┐ ┌─────────────────────────┐ │ Sink (JDBC Sink o app │ │ Monitoreo │ │ propia tipo central-sync)│ │ métricas + alertas reales│ └──────────┬───────────┘ │ (Slack / email / PagerDuty)│ ▼ └─────────────────────────┘ ┌─────────────────────┐ │ Base Central │ │ (PostgreSQL / Azure SQL) │ └─────────────────────┘

La diferencia con el laboratorio no es conceptual — es de escala y de quién administra qué. El laboratorio ya prueba, con datos reales, que este patrón (captura → Kafka → consolidación → auto-recuperación) funciona de punta a punta; lo que cambia en producción es la topología de red (sección 1), el tamaño del clúster Kafka, y el nivel de las garantías de seguridad.

3. Instalación y configuración por sucursal

Se repite igual en cada sucursal nueva — por eso vale la pena documentarlo una vez bien y automatizarlo (script de instalación o playbook de Ansible) antes de la sucursal número 10.

3.1 — Preparar el MySQL del POS

  1. Confirmar versión: MySQL 5.7+ u 8.0 (Debezium soporta ambas; se recomienda 8.0 si hay opción de elegir).
  2. Habilitar el binlog en formato ROW con imagen completa, y GTID — la misma receta validada en el laboratorio: 
    log_bin = ON
binlog_format = ROW
binlog_row_image = FULL
gtid_mode = ON
enforce_gtid_consistency = ON
  3. Ajustar la retención del binlog para dar margen de recuperación ante una sucursal desconectada por varios días (en el laboratorio no hace falta, pero en producción es crítico): 
    binlog_expire_logs_seconds = 604800  # 7 días
  4. Crear el usuario dedicado con privilegios mínimos (idéntico al de mysql-init/pos/01-schema.sql del laboratorio): 
    CREATE USER 'debezium'@'%' IDENTIFIED BY '<password fuerte, gestionada por el vault>';
GRANT SELECT, RELOAD, SHOW DATABASES, REPLICATION SLAVE, REPLICATION CLIENT
    ON *.* TO 'debezium'@'%';
  5. Verificar espacio en disco disponible para el binlog con la retención elegida.

3.2 — Instalar el agente Debezium Server

  1. Descargar la distribución oficial de debezium-server desde el proyecto Debezium, en el servidor de la sucursal (o un mini-PC/appliance junto al POS si el propio servidor de venta no debe correr software adicional).
  2. Configurar conf/application.properties: 
    debezium.source.connector.class=io.debezium.connector.mysql.MySqlConnector
debezium.source.database.hostname=localhost
debezium.source.database.port=3306
debezium.source.database.user=debezium
debezium.source.database.password=${vault:debezium/sucursalNN/db_password}
debezium.source.database.server.id=<único por sucursal — llevar registro central>
debezium.source.topic.prefix=pos.sucursalNN
debezium.source.table.include.list=pos_sucursalNN.medicamentos,pos_sucursalNN.ventas,...
debezium.source.snapshot.mode=when_needed

debezium.sink.type=kafka
debezium.sink.kafka.producer.bootstrap.servers=kafka-central.empresa.com:9093
debezium.sink.kafka.producer.security.protocol=SASL_SSL
debezium.sink.kafka.producer.sasl.mechanism=SCRAM-SHA-512
    Nota sobre snapshot.mode: en el laboratorio se usa initial (foto completa siempre al arrancar) porque las tablas son pequeñas. En una tabla de producción con años de historial, eso puede tardar horas y bloquear recursos — when_needed combinado con snapshots incrementales evita eso.
  3. Arrancar como servicio del sistema operativo (unidad systemd), no como proceso manual — para que sobreviva a un reinicio del servidor.
  4. Verificar: confirmar que el snapshot inicial llega al tópico correspondiente en el Kafka central, y que las métricas del agente (expone JMX/Prometheus) se ven en el monitoreo.

4. Requerimientos

Infraestructura central

  • KafkaClúster de mínimo 3 brokers, factor de replicación ≥3 (el laboratorio usa 1 solo broker — suficiente para demostrar, no para producción)
  • SinkKafka Connect JDBC Sink, o una versión productizada del patrón de central-sync (contenedorizada, con las mismas garantías de commit manual + reintento ya validadas)
  • Base CentralPostgreSQL o Azure SQL (según lo definido originalmente) — el laboratorio usa MySQL solo por simplicidad de la demo
  • MonitoreoPrometheus/Grafana u equivalente, con alertas reales (Slack, email, PagerDuty) conectadas a las señales que el laboratorio ya deja preparadas (ver sección 6)

Por sucursal

  • CómputoSuficiente para correr Debezium Server (un JVM liviano) — el propio servidor POS si tiene margen, o un mini-PC dedicado si no
  • RedSalida a internet (TLS, puerto configurado) hacia el Kafka central — no requiere IP fija ni puerto de entrada
  • MySQLBinlog habilitado, usuario dedicado, espacio en disco para la retención elegida

Seguridad

  • En tránsitoTLS obligatorio entre cada sucursal y el Kafka central (nunca texto plano en redes públicas)
  • SecretosGestor de secretos (Vault, AWS Secrets Manager, Azure Key Vault) — nada de contraseñas en archivos de configuración planos, a diferencia del laboratorio
  • PrivilegiosEl usuario de replicación mantiene el mismo principio mínimo ya validado: solo lectura de binlog, cero acceso de escritura a datos de negocio
  • RotaciónProceso periódico de rotación de credenciales, coordinado entre el vault y cada sucursal

Equipo y gobierno

  • CentralUn equipo de datos/plataforma responsable del clúster Kafka, el sink y el monitoreo
  • SucursalContacto de IT local (propio o de un proveedor) que pueda ejecutar la instalación inicial y responder ante una alerta de conectividad
  • Convención de IDsRegistro central de database.server.id y topic.prefix asignados — una colisión entre sucursales rompe la captura silenciosamente

5. Checklist de alta de una sucursal nueva

Pensado para repetirse 50+ veces sin sorpresas — candidato directo a automatizarse como playbook una vez validado a mano las primeras 2 o 3 veces.

  • Confirmar versión de MySQL y espacio en disco disponible
  • Habilitar binlog ROW + GTID, ajustar retención
  • Crear el usuario debezium dedicado, con password generada en el vault
  • Asignar server.id y topic.prefix únicos (verificar contra el registro central)
  • Instalar y configurar Debezium Server
  • Confirmar conectividad saliente (TLS) hacia el Kafka central
  • Arrancar el servicio y verificar el snapshot inicial en el tópico correspondiente
  • Confirmar que la sucursal aparece en el monitoreo y que las métricas fluyen
  • Hacer un cambio de prueba end-to-end y confirmar que llega a la Base Central

6. Del laboratorio a producción

El laboratorio no es solo una demo conceptual — ya probamos en vivo varios de los mecanismos de resiliencia que la versión de producción necesita. Esto es lo que ya está validado y lo que falta.

Ya validado en el laboratorio

  • Forma y contenido de los eventos CDC (before/after, tipos de dato, tombstones)
  • Consolidación de múltiples fuentes en una base central sin polling
  • Recuperación automática del sink ante una caída de la base destino (commit manual + reintento con backoff)
  • Recuperación automática de un conector caído (equivalente a connector-watchdog)
  • Heartbeat + healthcheck real del proceso de sincronización
  • Persistencia de los datos de Kafka ante un reinicio

Falta para producción real

  • Definir topología de red (Opción A vs B — sección 1)
  • TLS real + gestor de secretos (el laboratorio usa contraseñas en texto plano a propósito, es un entorno local)
  • Clúster Kafka con replicación ≥3 (el laboratorio corre 1 solo broker)
  • Snapshot incremental en vez de initial para tablas grandes
  • Motor destino final (Postgres/Azure SQL en vez de MySQL)
  • Monitoreo con alertas reales conectadas a un canal (hoy es solo una línea en el log)
  • Automatización del alta de sucursales (playbook, no pasos manuales)

7. Decisiones abiertas

Estas cuatro definiciones cambian bastante el diseño final — vale la pena resolverlas antes de automatizar el rollout.

¿Kafka propio o administrado?
Self-hosted (más control, más operación propia) vs. un servicio administrado tipo Confluent Cloud / Amazon MSK / Aiven (menos operación, costo recurrente distinto).
¿Motor de la Base Central?
PostgreSQL, Azure SQL, o mantener MySQL — afecta qué sink usar y qué equipo la administra.
¿Dónde corre Debezium Server en cada sucursal?
En el propio servidor del POS (más simple, pero comparte recursos con el sistema de venta) vs. un mini-PC/appliance dedicado junto a él (aísla el impacto, un costo de hardware adicional por sucursal).
¿Cómo se gestionan 50+ IDs y prefijos sin colisiones?
Un registro central simple (hoja de cálculo controlada) vs. generarlos automáticamente a partir de un catálogo de sucursales ya existente en el negocio.
