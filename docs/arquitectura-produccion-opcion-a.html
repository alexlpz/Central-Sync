De laboratorio a producción (Opción A): Kafka Connect centralizado

Cómo escalar el mismo patrón que ya corre en el laboratorio — un clúster de Kafka Connect en el centro, con un conector por sucursal, conectándose hacia cada MySQL de POS — a una red real de sucursales, asumiendo que la empresa ya cuenta con una VPN corporativa madura entre el centro y cada punto de venta.

Basado en: debezium-lab (el laboratorio ya es, arquitectónicamente, Opción A) Escenario objetivo: 50+ sucursales sobre VPN corporativa existente

1. Por qué Opción A en este caso

La decisión de fondo sigue siendo la misma que en la propuesta general: ¿quién se conecta hacia quién? Este documento desarrolla la rama que se deja de lado en la propuesta basada en Opción B.

  Opción A — Kafka Connect centralizado Opción B — Debezium Server por sucursal
Cómo funciona Un clúster de Kafka Connect en el centro corre un conector por sucursal (es el modelo exacto del laboratorio, con las 3 sucursales en la misma red Docker) Un proceso liviano (Debezium Server) instalado junto a cada MySQL de sucursal, que empuja los eventos directo al Kafka central
Dirección de la conexión El centro se conecta hacia cada sucursal — necesita una ruta de entrada a cada MySQL de POS Cada sucursal se conecta hacia afuera, al Kafka central — no requiere abrir ningún puerto de entrada
Firewall / NAT en la sucursal Requiere una ruta entrante permanente por sucursal — asumible sin fricción si ya existe una VPN corporativa, porque la ruta corre dentro del túnel, no contra internet abierto Tráfico saliente por HTTPS/TLS estándar — compatible con el firewall típico de una sucursal de retail sin cambios especiales
Ante una sucursal caída o sin internet El conector de esa sucursal queda FAILED en el centro; no hay agente local reintentando — la recuperación depende de que la sucursal vuelva y el centro reconecte (lo que connector-watchdog ya automatiza) El agente local sigue leyendo el binlog aunque no haya salida a internet, y reintenta el envío solo — tolera mejor una conexión intermitente
Cuándo tiene sentido Opción A es la elección correcta cuando ya existe una VPN corporativa madura conectando todas las sucursales al centro este documento — es más simple de operar (un solo lugar para administrar todos los conectores, nada que instalar ni mantener en cada sucursal) y es exactamente lo que el laboratorio ya modela.
Si la VPN corporativa no existe todavía o la conectividad de las sucursales es inestable/intermitente, conviene revisar la propuesta basada en Opción B (arquitectura-produccion.html) — ahí el agente local tolera mucho mejor una sucursal que se cae o pierde internet por horas.

2. Arquitectura objetivo

El centro concentra toda la captura: un clúster de Kafka Connect con un conector por sucursal, cada uno abriendo su propia conexión de replicación MySQL a través de la VPN. Nada se instala en la sucursal más allá de la configuración del propio MySQL.

┌─ Sucursal 1 ──────────────────┐ │ MySQL POS │ │ └─ binlog ROW + GTID │◄──────────────┐ │ └─ usuario dedicado │ │ └─────────────────────────────────┘ │ ┌─ Sucursal 2 ──────────────────┐ │ VPN / túnel site-to-site │ MySQL POS (binlog + usuario) │◄──────────────┼──────────┐ └─────────────────────────────────┘ │ │ ┌─ Sucursal N (50+) ─────────────┐ │ │ │ MySQL POS (binlog + usuario) │◄──────────────┘ │ └─────────────────────────────────┘ ▼ ┌───────────────────────────────┐ │ Kafka Connect (clúster de workers,│ │ 1 conector por sucursal, │ │ balanceados entre workers) │ └───────────────┬───────────────┘ │ ▼ ┌───────────────────────────────┐ │ Kafka (clúster, 3+ brokers, │ │ factor de replicación ≥3) │ └───────────────┬───────────────┘ ┌────────────────────┼────────────────────┐ ▼ ▼ ┌─────────────────────────┐ ┌─────────────────────────┐ │ Sink (JDBC Sink o app │ │ Monitoreo │ │ propia tipo central-sync) │ │ métricas + alertas reales│ └────────────┬────────────┘ └─────────────────────────┘ ▼ ┌─────────────────────────┐ │ Base Central │ │ (PostgreSQL / Azure SQL) │ └─────────────────────────┘

Esto no es una reinterpretación del laboratorio — es literalmente su misma topología, escalada. Lo que cambia entre el laboratorio y esta propuesta es: la VPN reemplaza a la red Docker compartida, el clúster de Kafka Connect pasa de un worker a varios (para repartir 50+ conectores), y el clúster de Kafka gana brokers y replicación.

3. Instalación y configuración por sucursal

A diferencia de Opción B, en cada sucursal nueva no se instala ningún agente — el trabajo real ocurre del lado del centro (sección 3.2). Lo único que toca a la sucursal es preparar su MySQL y habilitar la ruta de red.

3.1 — Preparar el MySQL del POS

  1. Confirmar versión: MySQL 5.7+ u 8.0 (Debezium soporta ambas; se recomienda 8.0 si hay opción de elegir).
  2. Habilitar el binlog en formato ROW con imagen completa, y GTID — la misma receta validada en el laboratorio: 
    log_bin = ON
binlog_format = ROW
binlog_row_image = FULL
gtid_mode = ON
enforce_gtid_consistency = ON
  3. Ajustar la retención del binlog para dar margen de recuperación ante un conector que quede FAILED varios días (por ejemplo, un corte largo de la VPN): 
    binlog_expire_logs_seconds = 604800  # 7 días
  4. Crear el usuario dedicado con privilegios mínimos (idéntico al de mysql-init/pos/01-schema.sql del laboratorio), permitiendo la conexión solo desde el rango de IP del clúster de Kafka Connect (no '%' como en el laboratorio): 
    CREATE USER 'debezium'@'10.x.x.x/mask-del-connect' IDENTIFIED BY '<password fuerte, gestionada por el vault>';
GRANT SELECT, RELOAD, SHOW DATABASES, REPLICATION SLAVE, REPLICATION CLIENT
    ON *.* TO 'debezium'@'10.x.x.x/mask-del-connect';
  5. Habilitar, dentro de la VPN, la ruta entrante desde la subred del clúster de Kafka Connect hacia el puerto de MySQL de esa sucursal (coordinado con el equipo de red).
  6. Verificar espacio en disco disponible para el binlog con la retención elegida.

3.2 — Registrar el conector en el Kafka Connect central

  1. Asignar database.server.id y topic.prefix únicos para la sucursal (registro central — una colisión rompe la captura de ambas sucursales en silencio, sin error explícito).
  2. Registrar el conector vía la API REST de Kafka Connect, siguiendo el mismo formato que connector/register-sucursalNN-connector.json del laboratorio, con el database.hostname apuntando a la IP/hostname de la sucursal dentro de la VPN: 
    {
  "name": "pos-sucursalNN-connector",
  "config": {
    "connector.class": "io.debezium.connector.mysql.MySqlConnector",
    "database.hostname": "10.x.x.x",
    "database.port": "3306",
    "database.user": "debezium",
    "database.password": "${vault:debezium/sucursalNN/db_password}",
    "database.server.id": "<único por sucursal>",
    "topic.prefix": "pos.sucursalNN",
    "table.include.list": "pos_sucursalNN.medicamentos,pos_sucursalNN.ventas,...",
    "snapshot.mode": "when_needed"
  }
}
    Nota sobre snapshot.mode: igual que en la propuesta de Opción B, el laboratorio usa initial porque las tablas son pequeñas; en producción con historial grande conviene when_needed + snapshots incrementales para no bloquear recursos horas.
  3. Verificar el estado del conector (RUNNING, sin tareas FAILED) y que el snapshot inicial llega al tópico correspondiente. connector-watchdog (o su equivalente productizado) debe estar vigilando este conector desde el momento en que se registra.
  4. Confirmar que las métricas de Kafka Connect (JMX/Prometheus) reflejan la nueva sucursal en el monitoreo central.

4. Requerimientos

Infraestructura central

  • Kafka ConnectClúster de varios workers en modo distribuido (no uno solo como en el laboratorio) — con 50+ conectores hay que dimensionar cuántos conectores por worker tolera cada JVM antes de degradar el rebalanceo
  • KafkaClúster de mínimo 3 brokers, factor de replicación ≥3 (el laboratorio usa 1 solo broker — suficiente para demostrar, no para producción)
  • VPNConcentrador/gateway VPN capaz de sostener 50+ túneles o rutas simultáneas, con el equipo de red como dueño de esa capa
  • SinkKafka Connect JDBC Sink, o una versión productizada del patrón de central-sync (contenedorizada, con las mismas garantías de commit manual + reintento ya validadas)
  • Base CentralPostgreSQL o Azure SQL (según lo definido originalmente) — el laboratorio usa MySQL solo por simplicidad de la demo
  • MonitoreoPrometheus/Grafana u equivalente, con alertas reales (Slack, email, PagerDuty) conectadas a las señales que el laboratorio ya deja preparadas (ver sección 6)

Por sucursal

  • CómputoNinguno adicional — a diferencia de Opción B, no hay agente que instalar ni mantener en la sucursal
  • RedRuta entrante habilitada dentro de la VPN, desde la subred del clúster de Kafka Connect hacia el puerto de MySQL — sin esto no hay Opción A posible
  • MySQLBinlog habilitado, usuario dedicado restringido a la subred de Connect, espacio en disco para la retención elegida

Seguridad

  • En tránsitoLa VPN cifra el enlace centro-sucursal, pero conviene TLS también dentro del túnel hacia MySQL (defensa en profundidad, no solo perímetro)
  • Secretos de conectoresKafka Connect guarda la configuración de cada conector (incluida la contraseña) en un tópico interno en texto plano por defecto — requiere un ConfigProvider (Vault, AWS Secrets Manager, Azure Key Vault) para no exponer 50+ contraseñas en ese tópico
  • PrivilegiosEl usuario de replicación mantiene el mismo principio mínimo ya validado: solo lectura de binlog, cero acceso de escritura a datos de negocio, y restringido por host a la subred de Connect
  • RotaciónProceso periódico de rotación de credenciales, coordinado entre el vault y el registro de conectores (rotar implica actualizar la config del conector vía REST, no solo la base)
  • Blast radiusUn clúster de Kafka Connect comprometido puede alcanzar las 50+ sucursales a la vez — a diferencia de Opción B, donde comprometer un agente solo expone una sucursal

Equipo y gobierno

  • CentralUn equipo de datos/plataforma responsable del clúster Kafka, el clúster de Kafka Connect (incluido su rebalanceo y escalado), el sink y el monitoreo
  • RedEquipo de redes responsable de la VPN y de aprobar/abrir cada ruta entrante nueva — es una dependencia dura para dar de alta cada sucursal
  • SucursalContacto de IT local solo para la preparación puntual del MySQL — sin responsabilidad operativa continua, a diferencia de Opción B
  • Convención de IDsRegistro central de database.server.id y topic.prefix asignados — una colisión entre sucursales rompe la captura silenciosamente

5. Checklist de alta de una sucursal nueva

Pensado para repetirse 50+ veces sin sorpresas. A diferencia de Opción B, casi todo el trabajo recurrente cae del lado del centro, no de la sucursal.

  • Confirmar versión de MySQL y espacio en disco disponible en la sucursal
  • Habilitar binlog ROW + GTID, ajustar retención
  • Crear el usuario debezium dedicado, restringido a la subred de Connect, con password generada en el vault
  • Solicitar y validar la ruta VPN entrante desde el clúster de Kafka Connect hacia el MySQL de la sucursal
  • Asignar server.id y topic.prefix únicos (verificar contra el registro central)
  • Registrar el conector vía la API REST del Kafka Connect central
  • Confirmar estado RUNNING y snapshot inicial en el tópico correspondiente
  • Confirmar que connector-watchdog (o equivalente) está vigilando el conector nuevo
  • Confirmar que la sucursal aparece en el monitoreo y que las métricas fluyen
  • Hacer un cambio de prueba end-to-end y confirmar que llega a la Base Central

6. Del laboratorio a producción

Con Opción A, el laboratorio no es una aproximación conceptual — es la misma arquitectura, sin cambiar de patrón. Lo que crece es la escala (workers, sucursales, brokers), no el diseño.

Ya validado en el laboratorio

  • Registro de conectores por sucursal contra la API REST de Kafka Connect, exactamente como se haría en producción
  • Forma y contenido de los eventos CDC (before/after, tipos de dato, tombstones)
  • Consolidación de múltiples fuentes en una base central sin polling
  • Recuperación automática del sink ante una caída de la base destino (commit manual + reintento con backoff)
  • connector-watchdog: recuperación automática de conectores FAILED y UNASSIGNED — el mecanismo que Opción A necesita en producción, ya probado en vivo
  • Heartbeat + healthcheck real del proceso de sincronización
  • Persistencia de los datos de Kafka ante un reinicio

Falta para producción real

  • VPN corporativa con 50+ rutas activas (el laboratorio usa una única red Docker compartida)
  • Clúster de Kafka Connect con varios workers y conectores distribuidos entre ellos (el laboratorio corre un worker con 3 conectores)
  • ConfigProvider para no exponer contraseñas de conectores en texto plano en el tópico interno de configuración
  • Clúster Kafka con replicación ≥3 (el laboratorio corre 1 solo broker)
  • Snapshot incremental en vez de initial para tablas grandes
  • Motor destino final (Postgres/Azure SQL en vez de MySQL)
  • Monitoreo con alertas reales conectadas a un canal (hoy es solo una línea en el log)
  • Automatización del alta de sucursales (playbook de registro de conector + ruta VPN, no pasos manuales)

7. Decisiones abiertas

Estas definiciones son específicas de operar Opción A a esta escala — vale la pena resolverlas antes de automatizar el rollout.

¿Kafka propio o administrado?
Self-hosted (más control, más operación propia) vs. un servicio administrado tipo Confluent Cloud / Amazon MSK / Aiven (menos operación, costo recurrente distinto). Con Opción A esto se extiende también a si Kafka Connect corre autoadministrado o como parte del mismo servicio administrado.
¿Cómo se dimensiona el clúster de Kafka Connect?
Cuántos workers y cuántos conectores máximo por worker antes de que el rebalanceo se vuelva lento — a diferencia de Opción B, este es un límite compartido por todas las sucursales, no un costo por sucursal.
¿Qué topología de VPN hacia las sucursales?
Hub-and-spoke clásico hacia el centro (más simple, todo pasa por el mismo concentrador) vs. una malla gestionada por el proveedor de red — condiciona qué tan rápido y con cuánta fricción se puede agregar una sucursal nueva.
¿Motor de la Base Central?
PostgreSQL, Azure SQL, o mantener MySQL — afecta qué sink usar y qué equipo la administra.
¿Cómo se gestionan los secretos de 50+ conectores?
Un ConfigProvider centralizado (Vault, AWS Secrets Manager, Azure Key Vault) integrado a Kafka Connect vs. inyección manual de credenciales al registrar cada conector — la primera opción es la única que escala sin exponer contraseñas en el tópico interno de configuración.
