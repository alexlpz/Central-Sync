Introducción

El punto débil: consultar el POS en vivo

Según el entendimiento de la última sesión: hoy el portal de Superlotengo consulta 4 bases de datos en vivo para armar lo que ve el usuario. El eslabón más débil es consultar directamente el POS de cada sucursal para inventario y costos — esas consultas son lentas, y esa latencia se traslada directo a la experiencia del usuario.

El objetivo: que Superlotengo deje de consultar el POS en vivo, y minimizar esa latencia para tener la información disponible de forma mucho más rápida.

La solución propuesta

Que el POS avise, no que Superlotengo pregunte

Cada MySQL de sucursal ya lleva su propio registro interno de cambios (el binlog). En vez de consultar el POS cada vez, leemos ese registro directo y mantenemos una copia local siempre al día — ahí es donde Superlotengo consulta.

● Medido en este laboratorio con Debezium Server standalone
ComponenteMedido (1 agente aislado, en reposo)Consideración en producción
Debezium Server (agente)~354 MB RAM · ~1% CPUUn proceso propio por sucursal — nada compartido, ninguna falla se propaga entre sitios
MySQL por sucursal~230 MB RAM · 2.2% CPUSin cambios — el mismo binlog que ya usa la sucursal
Kafka + Sincronización a Central~930 MB RAM · ~3% CPUUn único destino en el centro, sin importar cuántas sucursales se sumen
Red por sucursal—Saliente (HTTPS/TLS) — igual que el cobro con tarjeta, nunca se abre entrada

El costo total a escala completa es la suma de N agentes livianos (uno por sucursal) en vez de un clúster central compartido — pero cada agente pesa menos que un worker de Kafka Connect, y una falla queda aislada a una sola sucursal. Ambas mediciones son del mismo laboratorio, con 1 sucursal de prueba — no son carga de producción. El dimensionamiento real (memoria por agente, tamaño del clúster de Kafka en el centro) se define con una prueba de carga durante el piloto (Fase 1).
