Guion de demo para el cliente

Paso a paso de qué correr en terminal, qué hacer en DBeaver, y qué debería aparecer en cada momento — para presentar el laboratorio de CDC (3 sucursales → Kafka → Base Central) sin improvisar comandos en vivo.

Duración estimada: 25–35 min Terminales: 2 Requiere: DBeaver + Docker corriendo

0. Antes de que llegue el cliente

Todo esto se hace en privado, 10–15 minutos antes. No es parte de lo que ve el cliente.

1
Confirmar que el ambiente está sano
Terminal 
docker compose ps
Deberías ver Los 8 contenedores Up, y mysql-sucursal-01/02/03, mysql-central y central-sync marcados (healthy).
2
Confirmar que los 3 conectores están activos
Terminal 
for n in pos-sucursal-01-connector pos-sucursal-02-connector pos-sucursal-03-connector; do
  curl -s "http://localhost:8083/connectors/${n}/status" \
    | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['name'],'->',d['connector']['state'])"
done
Deberías verLos 3 en RUNNING. Si alguno no lo está, corre ./scripts/1-registrar-conector.sh antes de seguir.
3
Abrir las 4 conexiones en DBeaver
DBeaver
ConexiónHostPuertoBaseUsuario / pass
Sucursal 01localhost3306pos_sucursal_01root / rootpassword
Sucursal 02localhost3307pos_sucursal_02root / rootpassword
Sucursal 03localhost3308pos_sucursal_03root / rootpassword
Base Centrallocalhost3309central_farmaciasroot / rootpassword
Deja abiertas las 4 pestañas de SQL editor, una por conexión, listas para escribir — no las abras en vivo frente al cliente.
4
Acomodar la pantalla
Terminal A a la izquierda (para dejar corriendo 4-verificar-cambio.sh en vivo), DBeaver a la derecha. Terminal B la usas para lanzar los scripts sin tapar la vista de A.

Bloque A — Captura de cambios en tiempo real

El punto central del laboratorio: un cambio hecho directamente en el MySQL de una sucursal aparece en Kafka al instante, sin que nadie haya consultado esa base.

1
Dejar la terminal A viendo eventos en vivo
Terminal A 
./scripts/4-verificar-cambio.sh sucursal01 medicamentos
Deberías verUn mensaje "Viendo solo eventos NUEVOS..." y la terminal quieta, esperando.
2
Editar un producto en DBeaver
DBeaver — conexión Sucursal 01 
UPDATE medicamentos
SET precio_venta = 27.99, inventario = inventario - 10
WHERE sku = 'MED-003';
Di en voz alta "esto es exactamente lo que haría el sistema de punto de venta al hacer un ajuste de precio o una venta — un UPDATE normal, nada especial."
3
Señalar la Terminal A
Observar Terminal A
Deberías ver, en 1–2 segundos 
[UPDATE] id=3
  antes:   ... precio_venta=35.00 inventario=70
  despues: ... precio_venta=27.99 inventario=60
Qué significa Debezium leyó el binlog de MySQL (el registro interno de cambios que ya trae el motor), lo tradujo a un evento con el estado antes y después, y lo publicó en Kafka — nadie hizo polling a la base.

Bloque B — Ciclo de vida completo de un producto

Un caso más rico que un solo UPDATE: un producto que llega, se vende, cambia de precio y se da de baja — 5 eventos encadenados.

1
Lanzar la simulación en sucursal02
Terminal B 
./scripts/3-simular-cambios.sh sucursal02
El script pausa en cada paso ("Enter para continuar") — así puedes narrar cada uno antes de avanzar: alta de inventario, dos ventas (cada una con su fila en ventas/detalle_venta), una promoción, y finalmente la baja del producto.
2
(Opcional) Verlo reflejado en Terminal A
Terminal A 
./scripts/4-verificar-cambio.sh sucursal02 medicamentos
Deberías verUn [INSERT], varios [UPDATE] y un [DELETE] con SKU único (con timestamp) a medida que avanzas el script B.

Bloque C — Tres sucursales, una sola vista consolidada

Aquí se cierra la historia de negocio: 3 bases físicamente independientes, una Base Central que las refleja a todas sin que el portal central le pregunte nada a ninguna sucursal.

1
Hacer un cambio distinto en cada sucursal
DBeaver — sucursal01, sucursal02, sucursal03
Un UPDATE de precio en cada una (usa productos distintos por sucursal — revisa el catálogo de cada una si no te sabes los SKUs de memoria).
2
Abrir la conexión "Base Central" en DBeaver y refrescar (F5)
DBeaver — Base Central 
SELECT sucursal, sku, nombre, precio_venta, inventario, actualizado_en
FROM medicamentos
ORDER BY sucursal, sku;
Deberías verLos 3 cambios ya reflejados, cada fila con su columna sucursal — la vista consolidada de las 3 farmacias en una sola tabla.
Alternativa por terminal ./scripts/5-ver-central.sh hace la misma consulta sin salir de la terminal.

Bloque D — Resiliencia: se cae la Base Central

La pregunta que todo cliente hace tarde o temprano: "¿y si el servidor central se cae mientras las farmacias siguen vendiendo?" Este bloque lo responde en vivo.

Guion sugerido antes de empezar: "vamos a simular una caída real del servidor central — apagándolo de verdad, no con un truco. Van a ver que las sucursales ni se enteran, y que al recuperarlo todo se pone al día solo."
1
Correr el script de demo
Terminal B 
./scripts/6-demo-resiliencia.sh
Tiene 6 pausas narradas. En el paso 3 te va a pedir que hagas un cambio real en DBeaver (cualquier sucursal) — hazlo ahí mismo frente al cliente.
2
Señalar el healthcheck en el paso 4
Deberías ver 
Estado de salud de central-sync (Docker healthcheck): healthy
Qué significa El proceso sigue reintentando activamente contra la Base Central caída, y Docker mismo confirma que sigue vivo — no es solo "no se cayó", está funcionando de verdad.
3
Cerrar con la verificación en DBeaver
DBeaver — Base Central
El script termina pidiéndote refrescar (F5) la conexión a la Base Central — el cambio del paso 3 ya debe estar ahí, sin que nadie haya "reparado" nada a mano.

Bloque E — Resiliencia: se cae el conector de una sucursal

El segundo escenario de falla: no la Base Central, sino la pieza que lee los cambios de una sucursal específica.

Guion sugerido: "ahora el que se cae es el servidor de una sucursal puntual. Van a ver que el sistema lo detecta y lo reinicia solo, sin que nadie tenga que llamar a soporte a las 3am."
1
Correr el script de demo
Terminal B 
./scripts/7-demo-watchdog.sh sucursal02
6 pausas narradas: apaga el MySQL de esa sucursal, fuerza el fallo del conector, te muestra la causa exacta del error, y luego el reintento automático con backoff (30s, 60s, 120s…).
2
Levantar la sucursal cuando el script lo pida (paso 5)
El propio script hace el docker compose start — solo tienes que confirmar el Enter.
3
Cronómetro de recuperación (paso 6)
Deberías ver 
t+25s: RUNNING:RUNNING
>>> Recuperado en 25s, sin intervención manual <<<
Recuperación en 30–60 segundos típicamente, sin que nadie haya tocado la API de Kafka Connect a mano.

Bloque F — Cierre visual (opcional)

Si sobra tiempo, o si el cliente pide "ver Kafka por dentro" sin tecnicismos.

1
Abrir Kafdrop en el navegador
Navegador 
http://localhost:9000
Entra a cualquier tópico (ej. pos.sucursal01.pos_sucursal_01.medicamentos) → "View Messages" — útil para mostrar visualmente que cada cambio quedó como un mensaje individual, sin abrir terminal.

Al terminar

No hace falta apagar nada — el ambiente puede quedarse corriendo para la siguiente demo. Si necesitas dejarlo completamente limpio (catálogo original, sin los productos de prueba de 3-simular-cambios.sh): 
docker compose down -v
docker compose up -d --build
./scripts/1-registrar-conector.sh
